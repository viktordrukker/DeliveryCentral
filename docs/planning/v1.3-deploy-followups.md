## 🔴 Critical — open

### DEP-1 — Migration history bit-rot blocks fresh `prisma migrate deploy`
- **Severity:** Critical (blocks any non-baseline-aware DB init; blocks CI schema-drift gate)
- **Discovered:** 2026-04-26 during first VM deploy
- **Symptom:** `npx prisma migrate deploy` against an empty DB fails at `20260417_add_project_risks_enhanced_rag` with `ERROR: relation "project_rag_snapshots" does not exist`. The migration references a table created by a *later* migration.
- **Root cause:** Out-of-order FK reference + retroactive edit in migration history (already documented in repo memory `project-migration-history-bitrot.md`).
- **Workaround in v1.3.1:** `ops/bootstrap-app.sh` step 7 detects empty `_prisma_migrations`, drops/recreates schemas, applies `prisma/migrations/.baseline-schema.sql` as superuser, REASSIGNs ownership, marks all 92 migrations as applied via `prisma migrate resolve --applied`.
- **CI fallout:** `.github/workflows/ci.yml` "Schema drift check (DM-R-2)" job fails on every push because it runs the unworkarounded `prisma migrate deploy` from scratch.
- **Proper fix needed:** Reorder or consolidate the broken migrations so a fresh apply succeeds end-to-end. Options:
  - Squash all 92 migrations into a single baseline migration (loses history granularity).
  - Identify the specific out-of-order block and reorder timestamps + adjust FKs.
  - Either way: update `.baseline-schema.sql` and `.schema-hash` after the fix.
- **Next action:** Schedule a 1–2 day cleanup sprint. Without this, every fresh deploy needs the baseline workaround and CI stays red.

### DEP-2 — Schema-vs-code drift on `Skill.name` unique constraint
- **Severity:** Critical (blocks seeding without workaround; potential data-integrity surprise)
- **Discovered:** 2026-04-26 during seed run on prod DB
- **Symptom:** `prisma.skill.upsert({ where: { name } })` fails with `there is no unique or exclusion constraint matching the ON CONFLICT specification`.
- **Root cause:** `prisma/schema.prisma` declares `Skill.name String @unique` (single-column). The deployed schema (per baseline) has `@@unique([tenantId, name])` composite (added by `20260423_dm_7_5_7_skill_tenant_unique`). The seed code expects the single-column constraint.
- **Workaround in v1.3.1:** `bootstrap-app.sh` adds `skills_name_key UNIQUE (name)` constraint via superuser inside the baseline-apply path. Both constraints coexist (composite tenant-scoped + single-col).
- **Proper fix needed:** Decide which is correct. Composite tenant-scoped is right for multi-tenant; single-col is right for single-tenant. Then either:
  - Update `schema.prisma` to declare composite + remove single-col from DB via new migration, OR
  - Update later migration to keep single-col + remove composite.
  - Update seed to use composite where-clause if keeping composite.
- **Next action:** Open follow-up Prisma migration. Coordinate with whoever owns the tenant-isolation feature.

## 🟠 Important — open

### ~~DEP-3~~ — RESOLVED: chain of unrelated bugs uncovered + fixed
- **Status:** Resolved 2026-04-26
- **Discovered:** 2026-04-26 after pushing v1.3.1 + ssh-action repin
- **Root cause:** `ops/setup-github-secrets.sh` used `gh secret set NAME --body -` thinking `-` meant "read from stdin." It does NOT — `gh`'s CLI parser treats `-` as the literal string value. Result: every host/user/domain secret was set to a single `-` character. (Confirmed via `.github/workflows/debug-secrets.yml` probe: `actual_len=1, first_byte=2d` for all 6 secrets.)
- **Cascade of symptoms** (each masked the next):
  1. `dial tcp: lookup ***: no such host` — `-` is not a valid IP, fell through to DNS lookup; failed.
  2. `unknown option *** @` — `ssh "-@<host>"` parses `-@` as a flag.
  3. `ssh: handshake failed` — when SSH key was set the same wrong way, key body was `-`; auth never succeeded.
  4. **Plus an actually-real second bug:** `deploy.yml` only did `docker compose pull` (image-only) on the VM, never `git pull` of the working tree. So my v1.3.1 fix to `docker-compose.prod.yml` (adding SEED_* env passthrough) never reached the VM, and the container kept failing AppConfig validation `SEED_ADMIN_PASSWORD must be changed`.
  5. **Plus the original `appleboy/ssh-action` SHA was unresolvable** (commit was force-pushed/removed upstream); repinned to current `v1`.
- **Fixes shipped:**
  - **`ops/setup-github-secrets.sh`** (this commit): `set_secret` now pipes via `printf %s VAL | gh secret set NAME` (no `--body -`). SSH key uses `gh secret set NAME < $KEY_FILE` (stdin redirect for multi-line).
  - **`.github/workflows/deploy.yml`** (commits 4f11184, 4e81414, c0cb9ee, then this commit): `git fetch + git reset --hard origin/main` on every deploy syncs the compose file; switched from `appleboy/ssh-action` to `webfactory/ssh-agent + raw ssh` (better ED25519 support); SSH user hardcoded as literal `deploy` (not a secret anymore — saves one round-trip).
  - **All 6 corrupted secrets re-set** correctly via the fixed idiom (`printf %s VAL | gh secret set NAME`). Verified by re-running the debug workflow → all 6 now MATCH expected length and content.
  - **`.github/workflows/debug-secrets.yml`** deleted (one-shot diagnostic — purpose served).
- **Lesson for future-VM templates:** if you write a wrapper around `gh secret set`, NEVER use `--body -`. Always use the bare-pipe form (`echo VAL | gh secret set NAME`) or stdin redirect (`gh secret set NAME < file`) for clean values.

  - **Fix manually:** ~half-day to clean up the 225 remaining (mostly unused imports + `any`).
  - **Add lint to husky pre-commit:** prevents new debt from accumulating.
- **Next action:** Decide between baseline vs full cleanup. If baseline, write `scripts/eslint-baseline.json` similar to existing token baseline.

- **Severity:** Low (warning; deprecation in September 2026)
- **Symptom:** Every workflow run prints: *"Node.js 20 actions are deprecated. Will be forced to Node.js 24 by default starting June 2nd, 2026."*
- **Affected actions:** `actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683`, `actions/setup-node@49933ea5288caeca8642d1e84afbd3f7d6820020`.

### DEP-8 — `docker-compose.override.yml` files on VM are now redundant
- **Severity:** Low (cleanup)
- **Symptom:** `/opt/deliverycentral/docker-compose.override.yml` and `/opt/deliverycentral-staging/docker-compose.override.yml` exist with workarounds (`command: dist/src/main.js`, `AUTH_2FA_ENFORCE_ROLES: ""`, SEED_* env vars) that v1.3.1 baked into the image and base compose.
- **Risk:** Override files are NOT auto-discovered when GHA's `docker compose -f docker-compose.prod.yml ...` runs (the `-f` opts out). So they don't actively harm deploys — but they're confusing leftover state.
- **Proper fix:** After v1.3.1 successfully auto-deploys (DEP-3 resolved), delete both override files: `ssh deploy@VM "rm /opt/deliverycentral{,-staging}/docker-compose.override.yml"`.
- **Next action:** Defer until DEP-3 verified.

### DEP-9 — No automated backup-restore drill
- **Severity:** Medium (an untested backup is not a backup)
- **Symptom:** `ops/backup.sh` runs daily via cron and produces `*.sql.gz` files. `ops/restore-from-backup.sh` exists but only runs when invoked manually.
- **Proper fix:** Schedule monthly cron that runs `restore-from-backup.sh --target-db restore_drill_$(date +%Y%m)`, asserts row count, cleans up. Page on failure.
- **Next action:** Add to /opt/backups/ cron once a month is enough cadence given low data churn.
