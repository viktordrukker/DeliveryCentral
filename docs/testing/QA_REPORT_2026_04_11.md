## 2. FINDINGS — FUNCTIONAL ISSUES

### F-01: Director cannot access admin accounts API (P1)
- **Route:** GET /api/admin/accounts
- **Expected:** Director role should access account management
- **Actual:** 403 Forbidden — backend restricts to `admin` role only
- **Impact:** Director cannot view or manage user accounts from admin panel
- **Fix:** Add `director` to `@RequireRoles` on admin accounts endpoints

### F-02: Role-specific dashboards require personId query param (P2)
- **Route:** GET /dashboard/project-manager, /dashboard/resource-manager, /dashboard/hr-manager
- **Expected:** Should default to logged-in user's personId
- **Actual:** Returns 400 "Unsupported dashboard role" when called without personId
- **Impact:** Frontend handles this by passing personId from auth context; API is role-gated, not just param-gated
- **Note:** Working as designed — the admin role doesn't have PM/RM/HR dashboards. The frontend impersonation system handles this correctly.

### F-03: Login rate limit (10/min) blocks automated testing (P2)
- **Route:** POST /auth/login
- **Expected:** Reasonable rate for human users
- **Actual:** 429 after 10 logins across ALL accounts (IP-based, not per-account)
- **Impact:** Automated multi-role testing requires 60s cooldowns between batches
- **Suggestion:** Consider per-account rate limiting instead of per-IP for login, or exempt localhost

### F-04: Leave requests table was missing (FIXED)
- **Route:** GET /leave-requests/my
- **Was:** 500 Internal Server Error — table `leave_requests` didn't exist
- **Fix applied:** Migration `20260408_leave_requests` created and applied
- **Status:** RESOLVED

### F-05: Admin impersonation doesn't override backend API calls (P1)
- **Description:** The "View as" feature changes `useAuth().principal` on the frontend, but API calls still use the admin's JWT token. The backend validates roles from the JWT, not from the impersonation state.
- **Impact:** When impersonating an employee, the admin can still access admin endpoints (the backend sees the admin JWT). Role-specific dashboards (PM/RM/HR) will fail because the admin token doesn't carry those roles.
- **Suggestion:** Either: (a) send impersonated personId as a query param and let backend override, or (b) accept this as a frontend-only preview feature

## 3. FINDINGS — NON-FUNCTIONAL ISSUES

### NF-01: No HTTP request timeout was configured (FIXED)
- **Was:** No server.setTimeout() in main.ts
- **Fix applied:** `server.setTimeout(30000)` added
- **Status:** RESOLVED

### NF-02: No database connection pooling (FIXED)
- **Was:** Plain DATABASE_URL without pool params
- **Fix applied:** `connection_limit=25&pool_timeout=30` added to .env.example
- **Status:** RESOLVED (in .env.example; Docker containers use .env which may need updating)

### NF-03: Missing database indexes (FIXED)
- **Was:** 11 FK columns without indexes
- **Fix applied:** Migration `20260411_add_missing_indexes` with 11 CREATE INDEX statements
- **Status:** RESOLVED

### NF-04: Dashboard headcount trend fires sequential API calls (P2)
- **Description:** Main dashboard fires 12 sequential API calls to build headcount trend (1 per week)
- **Impact:** ~3-5 second load time for the trend chart
- **Suggestion:** Create a dedicated `/dashboard/workload/trend?weeks=12` endpoint that returns all weeks in one query

### NF-05: No caching layer (P3)
- **Description:** No Redis or in-memory cache for dashboard/master data queries
- **Impact:** Every dashboard load hits the database
- **Suggestion:** Add NestJS CacheModule with TTL-based caching for read-heavy endpoints

### NF-06: N+1 query patterns in dashboard services (P2)
- **Description:** RM, DM, and HR dashboard services loop through arrays finding related records with `.find()` instead of using Map-based lookups
- **Impact:** O(n²) complexity on large datasets
- **Fix:** Pre-build Map<id, record> from initial query results
