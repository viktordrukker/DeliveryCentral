# DM-R-25 — Point-In-Time Recovery runbook

**Goal:** restore the DB to any moment between the newest base backup and
the last archived WAL segment. RPO ≤ 60s (archive_timeout=60), RTO ≤ 15m
for a single-DB environment.

## What we have

- **Base backups** — weekly `pg_basebackup` into `.basebackups/<ts>.tar.gz`.
  Generated by `scripts/pg-basebackup.sh`. Retention: 4 weeks default.
- **WAL archive** — every WAL segment ≥ last base-backup is in
  `.wal-archive/`. Streamed there by the postgres container's
  `archive_command`. Retention: WAL files older than the oldest base
  backup can be pruned; keep at least two weeks.
- **Snapshots** (separate, DM-R-3) — coarse-grained restorable dumps.
  Prefer PITR when you need a surgical point-in-time.

## Recovery procedure

Two variables you need before starting: the target timestamp (or LSN),
and the base backup to use.

1. **Stop the backend + migrate containers** (DM-R-24 `panic-halt.sh` is
   the canonical first move):
   ```bash
   ./scripts/panic-halt.sh
   ```

2. **Pick a base backup** — the *most-recent* one whose timestamp is
   ≤ your target time.
   ```bash
   ls -1t .basebackups/*.tar.gz
   ```

3. **Stop Postgres + wipe the data directory** (destructive — confirm
   you have the snapshots + WAL):
   ```bash
   docker compose stop postgres
   docker volume rm deliverycentral_postgres-data
   ```

4. **Extract the base backup** into the volume:
   ```bash
   docker run --rm -v deliverycentral_postgres-data:/var/lib/postgresql/data \
     -v $(pwd)/.basebackups:/bb alpine sh -c \
     "tar -xzf /bb/<chosen-ts>.tar.gz -C /var/lib/postgresql/data --strip-components=1"
   ```

5. **Place a `recovery.signal` file + configure recovery parameters**:
   ```bash
   docker run --rm -v deliverycentral_postgres-data:/var/lib/postgresql/data alpine sh -c '
     touch /var/lib/postgresql/data/recovery.signal &&
     cat >> /var/lib/postgresql/data/postgresql.conf <<EOF
   restore_command = '\''cp /wal-archive/%f %p'\''
   recovery_target_time = '\''<YYYY-MM-DD HH:MM:SS UTC>'\''
   recovery_target_action = '\''promote'\''
   EOF
   '
   ```

6. **Start Postgres**. It will replay WAL up to the target time and
   then promote.
   ```bash
   docker compose up -d postgres
   docker compose logs -f postgres | grep -E 'archive|recovery'
   # Wait for: "database system is ready to accept connections"
   ```

7. **Verify**. `/api/health/deep` once backend is restarted; eyeball
   `migration_audit` to confirm the last row is ≤ target time;
   recompute hash chain via `node scripts/verify-audit-hash-chain.cjs`.

8. **Restart backend**:
   ```bash
   docker compose up -d backend
   ```

## Monthly restore drill (required)

Every month, on a throwaway instance, execute the above against a
randomly-selected target time in the last 7 days. Time each step;
measure the delta against RTO target. A missed drill is a Sev-2 incident.

DM-R-28 ties this drill into the quarterly chaos game-day; this monthly
cadence is the "known-answer" rehearsal between game-days.

## Common failures

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `FATAL: could not open file "pg_wal/0000…"` on startup | WAL segment missing from archive | Extend retention or rebuild archive from off-host copy (DM-R-27). |
| `recovery target is before the end of the last online backup` | Base backup is newer than target time | Pick an older base backup. |
| `FATAL: archive_command failed` during normal operation | `/wal-archive` volume full or RO | Make sure DM-R-27 off-host replication is draining the archive. |
